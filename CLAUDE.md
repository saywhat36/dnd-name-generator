# Fantasy Name Generator — CLAUDE.md

## Mission
A learning project to build production-style familiarity with Spring AI,
using Claude Code as the primary development partner. This repo has two
goals, not one:

1. Learn Spring AI's core features (ChatClient, structured output, tool
   calling, memory, RAG-adjacent techniques) by using each for a real
   feature, not a toy example.
2. Learn to work with Claude Code the way a production team would —
   clear specs, small reviewable changes, documented decisions, honest
   tradeoffs instead of manufactured justifications for features.

The product: a website where a user picks a fantasy race and gender and
gets back a list of names, drawn from a curated list and/or generated by
an LLM in a matching style.

## Current status
See `docs/ROADMAP.md` for phased plan and what's done.

## Architecture
See `docs/ARCHITECTURE.md` for full detail. Summary:
- Java 21, Spring Boot 3.x, Maven
- Spring AI for all LLM interaction — never call provider APIs directly
- Postgres, schema managed via Flyway (never hand-edit the schema)
- htmx + Thymeleaf frontend
- Single `names` table with a `source` column (CURATED / AI_GENERATED),
  not separate tables per source
- Shared (not per-user) pool of AI-generated names per race+gender combo,
  replenished asynchronously — requests never block on an LLM call
- Anonymous session ID (server-issued cookie) is the identity mechanism
  until Phase 2 auth exists

## Conventions
- Package by feature (`name/`, `generation/`, `favorite/`, `report/`),
  not by layer
- Prompts live in `src/main/resources/prompts/*.st`, never as inline
  Java strings
- Few-shot style examples are drawn **only** from `CURATED` rows, never
  from `AI_GENERATED` rows — this prevents the model imitating its own
  drifted output over time. This is a hard rule, not a preference.
- Structured LLM output → typed Java records via Spring AI's structured
  output converters. Never hand-parse model text.
- Deduplication of generated names is enforced by a DB unique constraint
  on `(normalized_name, race, gender)`. Application-level set-difference
  checks are an optimization on top of this, not the correctness
  mechanism — do not rely on app code alone to prevent duplicates.
- Never rely on LLM instructions ("don't repeat these names") as a
  correctness mechanism for anything — treat model output as untrusted
  and unreliable at following long negative constraints.
- Every new Spring AI feature gets a short entry in `docs/DECISIONS.md`
  explaining why and how it was wired in, including tradeoffs considered
  and rejected.
- Tests mock `ChatModel`/`ChatClient` at the service layer. Testcontainers
  for Postgres integration tests. No test should require a live API key
  to pass in CI.
- Don't add a Spring AI feature (tool calling, memory, RAG, etc.) just
  because a roadmap week names it — every feature must earn its place by
  solving an actual problem. If the natural solution is a plain DB query
  or a simple conditional, use that instead.
- Never use `JpaRepository.saveAll()` (or any JPA save path) for inserts
  that rely on `ON CONFLICT DO NOTHING` — JPA has no clean mapping for
  it, and a Postgres constraint violation poisons the whole transaction.
  Use the native insert path (`NameInsertDao`) for pool/favorite writes.
- Race, gender, source, and status are `varchar` + check constraints,
  never native Postgres enum types — see `docs/ARCHITECTURE.md` for why.
- Any `@Async` method must log its own outcome (success or failure) via
  an explicit try/catch with the log write in `finally` — do not assume
  Spring surfaces async exceptions on its own.
- The global LLM budget guard lives wherever the LLM is actually called
  (inside `PoolReplenishmentService`), not in the HTTP rate-limit filter.
  Don't assume a servlet filter sees async/background LLM traffic.

## Working with Claude on this repo
- Prefer small, single-feature changes matching one roadmap item at a time
- Before touching the service layer, prompt structure, or schema, propose
  the plan first and wait for confirmation
- Ask before adding new dependencies
- Push back if a request would violate a convention above rather than
  silently working around it — flag the tension explicitly
- Keep this file and `docs/ARCHITECTURE.md` updated if architecture or
  conventions change materially

## Do NOT
- Do not commit API keys — use environment variable placeholders in
  `application.yml`, keep local secrets in a gitignored file
- Do not hand-roll JSON parsing of model output
- Do not let AI-generated names skip the quality gate before entering
  the shared pool — it is permanent, shared, visible data
- Do not block a user-facing request on a live LLM call for name
  generation — always serve from the pool and replenish asynchronously
