# dnd-name-generator — Multi-Agent Review

_Generated 2026-07-08. Four parallel agents analysed the repo: feature ideation, security, code streamlining, and design-pattern correctness. No files were modified during the review._

## Headline

This is an unusually well-reasoned codebase for a learning project. The documented decisions match the code, hard conventions are followed with discipline, and there are no god services, circular dependencies, or SQL injection. The most material issues are a **broken authorization model on session-scoped data** (security) and a **budget guard that doesn't survive restarts** (architecture) — both worth fixing before this is meaningfully public. The biggest opportunity is realizing the unused Spring AI features (memory, streaming, evaluators) that the roadmap already reserves.

Effort: M. Must sit under the existing budget guard in `PoolReplenishmentService` and remain a filter, not the correctness mechanism.

### Priority 2 — strong value, distinct feature

4. **"Describe your character" intake** — free text (or an uploaded portrait) → `CharacterIntake(race, gender, styleHint)` via **structured-output classification / multimodality**. Must validate the model's race/gender against the enum set. Effort: M (text) / L (image).

5. **Second provider + cross-provider eval harness** — the roadmap's own Week 4 item; provider switching via profiles + evaluators as a test harness. Needs a second API key (ask first). Effort: M.

6. **Pronunciation guide** — cheap add-on structured shape, pairs with the etymology card. Effort: S–M.

### Priority 3 — interesting but weaker fit / higher risk

8. **MCP server** exposing the generator as a tool — weak product fit; only as a deliberate MCP exercise, documented honestly.

## 2. Security Review

Overall very security-aware. No SQL injection, no committed secrets, prompt-injection largely designed out. Key gaps below.

### HIGH

- **H1 — No real authorization on session-scoped data.** The session cookie is self-asserted; any valid UUID is trusted (the code comment admits it). Worse, `POST /names/{id}/flag` has **no session scoping at all** — any anonymous caller can flag any name, removing it from the pool (a content-integrity DoS). _Fix:_ gate `/flag` behind admin auth; sign the session cookie (HMAC) so fabricated/altered UUIDs are rejected.
- **H2 — No CSRF protection.** No Spring Security, no CSRF token. `POST /favorites`, `/reports`, `/names/{id}/flag`, `/browse/generate-more` rely solely on `SameSite=Lax`. `generate-more` triggers LLM spend. _Fix:_ add minimal Spring Security + CSRF for the htmx forms, or at least document Lax as the deliberate sole control (consider `SameSite=Strict` on mutations).

### MEDIUM

- **M1 — LLM cost controls are in-memory and partly bypassable.** `RateLimitFilter` is never wired up; `POST /browse/generate-more` is unauthenticated/unthrottled and can exhaust the 200/day budget in under a minute of scripted POSTs. _Fix:_ wire `RateLimitFilter` onto `generate-more`.
- **M2 — `/actuator/metrics` exposed without auth.** Leaks token-usage, timings, thread/session counts. _Fix:_ expose `health` only, or bind actuator to a management port / behind auth.
- **M3 — Full LLM prompt/response logging at DEBUG in all profiles** (`SimpleLoggerAdvisor`). Fine for today's data, risky once user text enters prompts (Phase 3). _Fix:_ gate to non-prod or INFP/off in prod.

### LOW

- **L1** — Session cookie lives 1 year with no server-side revocation (leaked cookie valid for a year).
- **L2** — Prompt-injection surface low and the CURATED-only few-shot convention is correctly followed; keep future refinement text out of prompt/example sections.
- **L3** — Model output is validated before insert (strict letter-only regex, parameterized DAO). Ensure the future `AI_REFINED` insert path reuses the same quality gate.
- **L4** — `.env` holds a live `GEMINI_API_KEY`, correctly gitignored and never committed. Rotate if the machine isn't trusted; add a placeholder `.env.example`.

**Done well:** parameterized native DAO, enum-typed input validation, secrets hygiene, cookie flags (HttpOnly/Secure/Lax), Thymeleaf auto-escaping (no `th:utext`).

**Priority order:** H1 (protect `/flag` + sign cookie) → H2/M1 (CSRF + wire rate limiter on `generate-more`) → M2 (actuator).

## 3. Code Streamlining (less code is better)

`src/main` is already lean; most "extra" code is load-bearing or documented intent. Realistic safe reduction: **~90–120 LOC, almost entirely in tests + a few dead getters.**

### Highest impact
1. **Shared `Name` mock fixture** — the `mock(Name.class)` + stub pattern repeats ~16× across tests. One `TestNames.name(id, displayName, source)` factory collapses each to a line (~30–40 LOC).
2. **Dedup ChatClient mock-chain setup** in `NameGenerationServiceTest` — route remaining tests through the existing `stubGenerationAttempts` helper (~15–20 LOC).

### Medium / low
3. **Narrow `NameGenerationService` public API** — `testPrompt`, `generateNameSuggestions(String)` and `(Race,Gender,int)` are only reached from tests; make them package-private (or drop `testPrompt` + its eval test).
4. **Remove clearly-unused `Name` getters** (`getNormalizedName`, `getProvider`, `getModel`, `getPromptVersion`, `getGenerationLogId`, `getCreatedAt`) — never read anywhere (~25–35 LOC). Keep `Favorite.getOwnerId` (reserved for Phase 2).

### Do NOT touch (documented intent, would violate conventions)
`RateLimitFilter` (staged infra), `AI_REFINED`/`REFINEMENT` enums (schema-synced), `NameInsertDao` native path (ON CONFLICT rule), explicit `@Query` methods (derived forms proven to throw), `AsyncConfig` executor/handler, `acceptedThisAttempt` multiset logic (simplifying reintroduces a documented bug).

## 4. Design-Pattern Correctness

Core patterns are implemented correctly and idiomatically. No god services, no circular deps, no leaky abstractions of consequence.

### Correct & idiomatic
- **Producer-consumer replenishment** — clean trigger/gate split, `@Async` self-invocation trap avoided, bounded `ThreadPoolTaskExecutor` correctly tuned, request path never blocks on the LLM.
- **DAO vs JpaRepository split** — `NameInsertDao` is the sole native `ON CONFLICT DO NOTHING` path; single-row idempotent JPA `save()`+catch used correctly for favorites/reports (the distinction is understood).
- **Structured output** — `.entity(ParameterizedTypeReference<List<NameSuggestion>>)`, zero hand-parsing anywhere. Advisor pattern used correctly; `ChatMemoryAdvisor` correctly absent (deferred).
- **Package-by-feature, constructor DI, records/factories** — exemplary.
- **Anemic `Name` entity** — present but deliberate and defensible (read-only projection; all writes via DAO). `GenerationLog` carries readable static factories.

### Needs work
- **1b — Budget guard not restart-safe.** In-memory counter resets to 0 on every restart; on the Render free tier (frequent cold starts/redeploys) a crash-loop blows past 200/day. _Fix:_ persist a `generation_budget(day, count)` row with atomic `UPDATE ... RETURNING`.
- **7b — `TaskRejectedException` can fault a DB-only request.** `NameService.getNames` calls `replenish` with no try/catch; queue saturation throws synchronously at the submission site into a user-facing read. _Fix:_ swallow-and-log (replenishment is best-effort).
- **7a — `saveSkip` pollutes `parse_success`.** Operational skips (budget/cap/no-examples) route through `GenerationLog.parseFailure`, corrupting any parse-failure-rate query. _Fix:_ add a distinct skip/outcome dimension.
- **4b (low)** — `DeduplicationService.normalize` lives in `generation` but is a `name` concern (called from `NameInsertDao`); consider relocating.

**Prioritized:** persist budget counter → guard `replenish` against `TaskRejectedException` → fix `parse_success` overload → (low) relocate `normalize`.

## Cross-cutting themes (where reviews agree)

- **`RateLimitFilter` should be wired up** — flagged by both security (M1) and features (the intended trigger for synchronous LLM endpoints). It's staged, not dead.
- **The budget guard is the safety net but is fragile** — security calls it "the saving grace" against cost DoS; architecture shows it doesn't survive restarts. Persisting it strengthens both.
- **Future synchronous LLM features (refinement, etymology, intake)** must register the rate limiter and keep user text out of prompt/example sections — features, security, and patterns all converge here.
- **Convention discipline is real** — CURATED-only few-shot, native-DAO-only inserts, externalized prompts, DB-constraint-as-correctness are all actually honored in code, not just docs.
